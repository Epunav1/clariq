🎯 Clariq MVP Technical Roadmap

Building the Predictive Revenue Operating System
6-Week Implementation Plan (May - June 2026)

What we're building: An AI engine that predicts what's happening in the market 12+ hours ahead of competitors, then automatically executes the perfect response to maximize seller revenue.

Why it matters: Most tools show data. Clariq makes money. Every seller should make $2-5K extra per month from our predictive execution.

The goal: 50 beta users by end of Week 6. Each user should see measurable revenue impact ($500-1000 minimum) from predictive automation.

📋 The Big Picture: System Architecture

Clariq is built in 4 interconnected layers. Each week focuses on one layer:

Layer 1: Data Ingestion & Real-Time Sync

Pulls Shopify data 24/7. Tracks every product, order, customer, price change. We collect data; competitors react to it. (Week 1-2)

Layer 2: Market Intelligence Network

Aggregate anonymous data from all Clariq sellers to spot patterns. When 50 sellers in a category make moves, we detect it and alert YOU before competitors do. (Week 2-3)

Layer 3: Predictive AI Engine

ML models predict: price war incoming, demand surge, inventory aging, churn risk. Predictions are 12+ hours ahead of market. (Week 3-4)

Layer 4: Autonomous Execution Engine

Once predicted, Clariq auto-executes: price updates, campaigns, emails, promotions. User approves first 10 times, then fully autonomous. (Week 4-6)

📅 Week-by-Week Breakdown

1
Foundation: Real-Time Data Ingestion

Goal: Build a real-time Shopify data pipeline that never sleeps

✓Shopify OAuth Flow (Improved)
- Add "Products", "Orders", "Customers" permission scopes (not just basic auth)
- Store access token encrypted in database
- Build token refresh mechanism (tokens expire every 24hrs)
- Test with 3 real Shopify stores
✓Real-Time Webhook Setup
- Listen to: product updates, price changes, order creation, inventory changes
- Store webhook events in queue (Redis or Celery)
- Don't process immediately (we'll batch process)
- Log every event for debugging
✓Data Models (PostgreSQL Schema)
Create tables for: Products, Prices, Orders, Customers, InventoryChanges
Each row includes timestamp (critical for predictions)
-- Key tables needed: CREATE TABLE products ( id UUID PRIMARY KEY, store_id UUID, shopify_id VARCHAR, name VARCHAR, current_price DECIMAL, created_at TIMESTAMP, updated_at TIMESTAMP ); CREATE TABLE price_history ( id UUID PRIMARY KEY, product_id UUID, old_price DECIMAL, new_price DECIMAL, changed_by VARCHAR, -- 'user' or 'clariq' reason VARCHAR, created_at TIMESTAMP ); CREATE TABLE orders ( id UUID PRIMARY KEY, store_id UUID, customer_id UUID, total_revenue DECIMAL, items_count INT, created_at TIMESTAMP );
CTO/Lead Dev: Build Shopify API integration from scratch. This is mission-critical.
Full-Stack: Database schema + migration scripts
DevOps: Set up Redis for event queue (if not already done)
⚠️ Important Note: Don't try to build everything in Week 1. Focus ONLY on: OAuth → Webhooks → Store Raw Data. If data isn't flowing reliably, everything else breaks.
✓ Success Criteria:
- Data flowing from 3 test stores every 10 seconds
- Zero missed webhook events (log and verify)
- Database has 1000+ historical price changes
- Latency from Shopify change to database: <2 seconds
2
Market Intelligence: The Collective Brain

Goal: Build the network effect—aggregate all seller data to see patterns no individual can

✓Anonymous Data Aggregation Pipeline
- Create aggregated tables (no store names, just anonymized data)
- By hour: "In category X, average price change was -5%, 200 products changed"
- Track patterns: discounts, bundles, seasonal shifts
- Privacy first: No way to identify individual sellers
✓Competitor Price Tracking (Simulated)
- For MVP: Use Clariq's own stores as "competitors"
- Track price changes across all stores by category
- Build "price war detection" (3+ competitors drop price within 1hr = price war)
- Show dashboard alert: "Price war detected in Electronics - 5 competitors moved"
✓Category-Specific Pattern Detection
- Electronics: Detect tech release cycles (iPhone launch = pricing opportunity)
- Fashion: Detect seasonal shifts (winter to spring = new styles)
- Build a simple rule engine: IF [pattern] THEN [alert]
-- Aggregated insights table: CREATE TABLE category_insights ( id UUID PRIMARY KEY, category VARCHAR, hour TIMESTAMP, avg_price_change DECIMAL, -- -5% means prices dropped avg 5% price_war_detected BOOLEAN, num_competitors_moved INT, demand_signal VARCHAR, -- 'high', 'normal', 'low' recommended_action VARCHAR, created_at TIMESTAMP );
Data Scientist: Build aggregation pipeline + pattern detection
Full-Stack: Build API endpoint to show alerts on dashboard
PM: Define what patterns matter most (price wars vs seasonal)
💡 Founder Note: This is where the real competitive advantage starts. Competitors track their own store. We track the entire market. By Week 2 end, you should see market patterns that are actually useful.
✓ Success Criteria:
- Dashboard shows 5+ actionable alerts per day
- Price war detected within 30min of happening
- Alerts are useful (not false alarms)
- Historical data shows clear patterns (e.g., "Tuesday discounts happen 60% of time")
3
Predictions: Seeing 12 Hours Into The Future

Goal: Build ML models that predict what WILL happen, not just what DID happen

✓Time-Series Forecasting (ARIMA / Prophet)
- For each seller: Predict demand 24 hours ahead
- Model: Historical sales data → predict tomorrow's sales
- Update model every 4 hours with new data
- Accuracy target: 85%+ for 24-hour forecast
✓Inventory Aging Prediction
- For each product: Predict if it will age out (not sell) in next 7 days
- Model inputs: Current age, sell-through rate, category trends
- Output: "This product will age out in 4 days unless you take action"
✓Price Elasticity Model
- For each product: Learn "how sensitive is demand to price change?"
- Some products: -5% price = +30% demand (elastic)
- Others: -5% price = +5% demand (inelastic)
- Use this later for optimization
-- ML Predictions table: CREATE TABLE predictions ( id UUID PRIMARY KEY, product_id UUID, prediction_type VARCHAR, -- 'demand_24h', 'inventory_aging', 'price_elasticity' predicted_value DECIMAL, confidence DECIMAL, -- 0.85 = 85% confident forecast_for TIMESTAMP, created_at TIMESTAMP );
Data Scientist: Build all 3 ML models (ARIMA, inventory aging, elasticity)
ML Eng (if available): Deploy models, set up inference pipeline
Full-Stack: Store predictions in DB, build API to retrieve
⚠️ Challenge: Cold start problem: New sellers have no historical data. Use industry averages until we have 2 weeks of data. Better to be approximate than no prediction.
✓ Success Criteria:
- Demand prediction: 85%+ accurate 24-hour ahead
- Inventory alerts trigger correctly 90% of time
- Price elasticity model captures real behavior
- Predictions update every 4 hours with new data
4
The Execution Engine Begins

Goal: Move from predictions to actions. Build the autopilot.

✓Price Optimization Engine
- Based on: prediction, elasticity, margin targets, competitor prices
- Calculate: "What price maximizes revenue AND margin?"
- Safety checks: Never drop below cost, never raise above 2x cost
- Initial: Require user approval. Later: Full auto
✓Approval & Execution Flow
- Generate recommended action: "Drop price to $24.99 (from $29.99). Reason: Demand spike predicted, elasticity high"
- Show user: Action + reason + estimated impact
- User clicks: Approve / Modify / Reject
- Log everything for learning
✓Shopify API Write Access
- Implement: Update product prices on Shopify automatically
- Safety: Batch updates (not instant), with confirmation
- Rollback: If something goes wrong, auto-revert to previous price
-- Execution recommendations table: CREATE TABLE execution_recommendations ( id UUID PRIMARY KEY, store_id UUID, product_id UUID, action_type VARCHAR, -- 'price_update', 'promotion', 'campaign' recommended_value VARCHAR, reason TEXT, estimated_impact VARCHAR, -- '+$500 revenue', '+15% demand' status VARCHAR, -- 'pending', 'approved', 'executed', 'rejected' user_action_at TIMESTAMP, executed_at TIMESTAMP, created_at TIMESTAMP );
CTO/Lead Dev: Price optimization algorithm + Shopify write API
Full-Stack: Approval UI + workflow management
PM: Define business rules (safety checks, margin targets)
⚠️ CRITICAL SAFETY MEASURE:
- Week 4: ALL actions require user approval (no auto-execution yet)
- Week 5: Approve 10 actions manually, then 20% can auto-execute
- Week 6: If working perfectly, full auto (but user can pause anytime)
✓ Success Criteria:
- Price recommendations generated 10+ times per day
- Each recommendation shows clear reason + impact estimate
- User approves 70%+ of recommendations (means they're good)
- Zero price update failures to Shopify
- Estimated revenue impact visible on dashboard

⚠️ Risks & How We Mitigate Them

Risk: Shopify API rate limits (we hit them and crash)
Mitigation: Build request queuing + exponential backoff. Cap at 400 requests/minute (well below Shopify's limits). Monitor daily.
Risk: Cold start problem (new users have no historical data for predictions)
Mitigation: Use industry benchmarks for first 2 weeks. "Your category average is X; you're Y% above/below." Improve as data accumulates.
Risk: Price update bugs (we crash a seller's pricing)
Mitigation: Week 4-5: All manual. Week 6: 20% auto with instant rollback capability. Never auto-execute without user having approved it manually first.
Risk: ML models are wrong (predictions are garbage)
Mitigation: Track prediction accuracy daily. If <70%, revert to simpler model. Better to be simple + accurate than complex + wrong.
Risk: Integration with SendGrid/Twilio fails
Mitigation: Build gracefully degraded: If SMS fails, try email. If email fails, show alert on dashboard instead. Never silently fail.
Risk: We run out of time in Week 6
Mitigation: CUT features progressively. In priority order (drop first if needed): Campaigns → Email automation → SMS → Full auto. But keep price optimization (it's the core).
